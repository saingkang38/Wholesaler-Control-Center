# 변경 대응 전략 문서
버전: 1.0
상태: 운영 기준 확정본

## 문서 목적
이 문서는 변경 요청이 들어왔을 때 변경 규모를 분류하고,  
수정 범위를 어디로 한정할지 판단하는 기준을 정의한다.

목표는 한 가지다.  
**변경이 생겨도 작은 범위에서 끝나게 만드는 것.**

---

## 1. 기본 원칙

1. 수정은 가능한 한 국소 범위에서 끝나야 한다.
2. 도매처 문제는 해당 collector 내부에서 먼저 해결한다.
3. 출력 양식 문제는 해당 exporter 내부에서 먼저 해결한다.
4. 비교 규칙 문제는 compare 계층에서 먼저 해결한다.
5. 인증/권한 문제는 auth/users 계층에서 먼저 해결한다.
6. 공통 계층 수정은 예외적으로만 허용한다.
7. 급한 패치보다 구조 유지가 우선이다.
8. 현재 단계 범위를 넘는 구조 변경은 바로 하지 않는다.
9. 토큰 절약을 위해 전체 파일 일괄 수정은 최후 수단으로 둔다.

---

## 2. 변경 규모 분류

### A. 소규모 변경
특정 모듈 내부 수정으로 끝나는 변경

#### 예시
- 특정 도매처 로그인 selector 변경
- 페이지 버튼 selector 변경
- 도매처 상세 DOM 일부 변경
- 출력 컬럼명 1~2개 변경
- 상태 텍스트 표현 변경
- 파일명 규칙 수정

#### 대응
- 해당 모듈 내부만 수정
- 공통 계층 수정 금지
- 영향 범위 확인 후 수정

---

### B. 중규모 변경
한 기능 영역의 구조를 조금 재정비해야 하는 변경

#### 예시
- 한 도매처의 목록 수집 방식 전체 변경
- compare 분류 기준 수정
- export 포맷 생성 흐름 개편
- 로그인 방식 고도화
- 실행 로그 구조 개선
- scheduler 실행 구조 개선

#### 대응
- 해당 기능 영역 단위 리팩터링
- 관련 문서 수정
- 테스트 갱신

---

### C. 대규모 변경
데이터 모델 또는 시스템 구조에 영향을 주는 변경

#### 예시
- 비교 기준이 `도매처 + 상품코드`에서 다른 기준으로 바뀜
- 단일 사용자 구조가 다중 조직 구조로 바뀜
- 엑셀 중심 비교에서 DB 중심 규칙 엔진으로 전환
- 외부 솔루션 양방향 연동 추가
- 권한 체계 대폭 확장
- 핵심 저장 방식 전환

#### 대응
- PRD / Phase / Schema부터 수정
- 부분 패치 금지
- 구조 재검토 후 진행

---

## 3. 변경 처리 순서

모든 변경은 아래 순서로 처리한다.

1. 변경 이유 확인
2. 변경 규모 분류
3. 영향 범위 확인
4. 수정 대상 모듈 지정
5. 문서 반영 여부 판단
6. 코드 수정
7. 테스트
8. 결과 검토

이 순서를 건너뛰고 바로 큰 수정부터 하지 않는다.

---

## 4. 변경 유형별 우선 수정 위치

### 도매처 수집 문제
우선 위치:
- `collectors/{wholesaler_name}/`

공통 수정 허용 조건:
- 2개 이상의 collector가 동일 문제를 공유할 때만

---

### 비교 판정 문제
우선 위치:
- `compare/`
- compare rule 계층
- compare service

공통 product model 변경은 예외적으로만 허용

---

### 출력 양식 문제
우선 위치:
- `exporters/{format_name}/`

공통 export engine 수정은  
파일 생성 공통 로직 문제일 때만 허용

---

### 사용자/로그인 문제
우선 위치:
- `auth/`
- `users/`

collector나 exporter와 섞어서 수정하지 않는다.

---

### 자동 실행 문제
우선 위치:
- `scheduler/`
- 실행 orchestration 계층

collector 로직과 스케줄 로직은 분리 유지

---

### 설정값 문제
우선 위치:
- `settings/`
- `CONFIG_POLICY.md`

코드 수정 전에 설정 분류부터 다시 판단한다.

---

## 5. 리팩터링 판단 기준

아래 중 하나라도 해당하면 리팩터링 검토 대상이다.

- 같은 유형의 조건문이 3회 이상 반복됨
- 특정 파일이 과도하게 커짐
- 한 함수가 로그인/수집/정규화/저장을 동시에 수행함
- 도매처별 분기가 중앙 파일에 늘어남
- exporter가 포맷별 분기문으로 누적됨
- compare 규칙이 여러 위치에 중복됨
- 사용자 실행 절차가 지나치게 복잡해짐

---

## 6. 재설계 판단 기준

아래 중 하나라도 해당하면 부분 패치보다 재설계를 우선 검토한다.

- 기본 비교 기준 자체가 바뀜
- 권한 구조가 크게 바뀜
- 다중 회사 / 다중 워크스페이스 개념이 추가됨
- 단일 실행 구조가 queue 기반으로 바뀌어야 함
- 데이터 저장 중심이 크게 바뀜
- 수집 방식이 브라우저 자동화 중심에서 API 중심으로 전환됨
- 운영 모델 자체가 달라짐

---

## 7. 절대 금지

1. 도매처별 예외를 공통 서비스 파일에 누적
2. 긴급 수정이라는 이유로 하드코딩 추가
3. 비교 규칙을 UI 코드에 배치
4. 출력 양식 변경을 controller에서 직접 처리
5. 로그인/권한 처리를 collector에 혼합
6. 변경 이유 문서화 없이 구조를 뜯음
7. 다음 단계를 미리 당겨와서 현재 구조를 흔듦

---

## 8. Claude 판단 규칙

Claude는 변경 요청을 받으면 먼저 아래를 답해야 한다.

1. 소규모 / 중규모 / 대규모 중 무엇인가
2. 왜 그렇게 판단했는가
3. 수정 대상 모듈은 어디인가
4. 공통 계층 수정이 필요한가
5. 문서 수정이 필요한가

이 판단 없이 바로 코드부터 수정하지 않는다.

---

## 9. 비개발자 진행 규칙

변경 요청을 처리할 때는 아래 형식을 우선한다.

1. 이번 변경 목표
2. 수정 범위
3. 사용자가 직접 해야 할 일
4. Claude가 제공할 결과물
5. 확인 방법

설명만 길게 하지 않는다.

---

## 10. 운영 원칙

1. 급해서 일단 넣는 방식 금지
2. 모듈 경계를 깨는 수정 금지
3. 자주 바뀌는 규칙은 설정 또는 별도 rule 파일로 이동
4. 한 번 수정한 문제는 재사용 가능한 방식으로 정리
5. 토큰 낭비를 유발하는 광범위 수정은 피함
