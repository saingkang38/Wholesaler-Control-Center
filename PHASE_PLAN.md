# 도매처 통합 관리 시스템 단계별 개발 계획
버전: 1.0
상태: 운영 기준 확정본

---

## 1. 전체 원칙

1. 한 번에 전체를 만들지 않는다.
2. 먼저 1개 도매처 파일럿을 완성한다.
3. 구조 안정화가 UI보다 우선이다.
4. 문서 확정 후 구현한다.
5. 단계 밖 구현은 하지 않는다.
6. 각 단계는 비개발자가 직접 확인 가능한 완료 기준을 가져야 한다.
7. Claude Code는 현재 단계에 필요한 범위만 다룬다.
8. 토큰 절약을 위해 다음 단계 구현은 미리 하지 않는다.

---

## 2. 단계 진행 형식

각 단계는 반드시 아래 형식으로 진행한다.

1. 현재 단계 목표
2. 이번 단계에서 Claude가 할 일
3. 이번 단계에서 사용자가 할 일
4. 사용자가 실행 후 보내줘야 할 결과
5. 완료 기준
6. 다음 단계 진입 조건

이 형식을 벗어나 장황하게 진행하지 않는다.

## Phase 0. 프로젝트 초기 세팅

### 목표
기본 실행 구조와 개발 기반을 만든다.

### Claude가 할 일
- 프로젝트 기본 폴더 구조 설계
- 환경변수 예시 파일 구조 제안
- 기본 웹앱 실행 구조 제안
- 로그인 화면 뼈대 제안
- 대시보드 뼈대 제안

### 사용자가 할 일
- 프로젝트 폴더 생성
- 기본 의존성 설치
- 로컬 실행 확인
- 환경변수 파일 준비

### 사용자가 보내줘야 할 결과
- 실행 성공 여부
- 에러 메시지
- 폴더 구조 캡처 또는 목록
- 실제 실행 화면 캡처

### 완료 기준
- 로컬 실행 가능
- 기본 라우팅 가능
- 환경변수 로드 가능

### 다음 단계 진입 조건
- 프로젝트가 최소 1회 정상 실행됨

---

## Phase 1. 사용자 인증 구조

### 목표
관리자/직원 계정 구조와 인증 흐름을 만든다.

### Claude가 할 일
- 사용자 테이블 구조 제안
- 로그인/로그아웃 흐름 제안
- 관리자/직원 역할 구조 제안
- 기본 세션 처리 구조 제안

### 사용자가 할 일
- DB 반영
- 로그인 실행 확인
- 계정 생성/로그인 테스트
- 화면 이동 테스트

### 사용자가 보내줘야 할 결과
- 로그인 성공 여부
- 권한 구분 동작 여부
- 발생한 에러 메시지
- DB 생성 여부

### 완료 기준
- 관리자/직원 로그인 가능
- 로그인 후 대시보드 접근 가능
- 비로그인 사용자는 보호된 화면 접근 불가

### 다음 단계 진입 조건
- 인증 흐름이 1회 이상 정상 동작함

---

## Phase 2. 수집 공통 골격

### 목표
모든 collector가 공통으로 따를 실행 구조를 만든다.

### Claude가 할 일
- collector 표준 인터페이스 설계
- collection run 구조 설계
- execution log 구조 설계
- 수동 실행 orchestration 구조 설계

### 사용자가 할 일
- 공통 골격 파일 생성
- 수동 실행 버튼 테스트
- 로그 저장 확인
- 빈 collector 테스트

### 사용자가 보내줘야 할 결과
- 수동 실행 결과
- run 생성 여부
- 로그 저장 여부
- 발생한 에러 메시지

### 완료 기준
- collector 추가 기준이 정해짐
- 수집 실행 이력이 저장됨
- 수동 실행 흐름이 동작함

### 다음 단계 진입 조건
- 실제 도매처 collector를 꽂을 준비가 끝남

---

## Phase 3. 오너클랜 파일럿 collector

### 목표
오너클랜 1개 도매처를 끝까지 성공시킨다.

### Claude가 할 일
- ownerclan collector 구조 제안
- 로그인 처리 구조 제안
- 목록 수집 구조 제안
- raw → normalized 흐름 제안
- 실패 로그 구조 제안

### 사용자가 할 일
- 실제 계정/환경 설정
- 로그인 테스트
- 수집 실행 테스트
- 다중 페이지 테스트
- 실패 케이스 확인

### 사용자가 보내줘야 할 결과
- 로그인 성공 여부
- 수집 성공/실패 로그
- 저장된 데이터 일부
- 에러 메시지
- 페이지별 수집 결과

### 완료 기준
- 로그인 가능
- 여러 페이지 수집 가능
- 상품코드 기준 저장 가능
- 성공/실패 로그 확인 가능

### 다음 단계 진입 조건
- 오너클랜 파일럿이 최소 기준 충족

### Claude가 할 일
- wholesaler 구조 제안
- normalized_products 저장 구조 제안
- 통합 상품 목록 화면 구조 제안
- 기본 검색/필터 구조 제안

### 사용자가 할 일
- 저장 확인
- 목록 화면 테스트
- 검색/필터 테스트
- 샘플 데이터 검증

### 사용자가 보내줘야 할 결과
- 저장된 데이터 화면 캡처
- 검색/필터 동작 여부
- 목록 에러 여부

### 완료 기준
- 수집 결과가 DB에 저장됨
- 목록 화면에서 조회 가능

### 다음 단계 진입 조건
- 최소 1개 도매처 데이터가 조회 가능

## 3. 단계 통제 규칙

1. 현재 Phase 완료 전 다음 Phase 핵심 구현을 시작하지 않는다.
2. 다음 Phase 준비용 문서 정리는 가능하지만 본 구현은 하지 않는다.
3. 범위가 커지면 기능을 줄여서라도 현재 Phase 완료 기준을 맞춘다.
4. 1차는 오너클랜 파일럿 성공이 최우선이다.
5. 비개발자가 실행할 수 없는 거대한 단계로 합치지 않는다.
6. 토큰 절약을 위해 현재 단계와 무관한 파일 전체 수정은 피한다.
