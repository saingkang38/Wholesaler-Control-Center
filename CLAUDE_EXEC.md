# 실행지침

## 8. Claude 작업 원칙

1. 현재 요청 범위만 다룬다. 요청하지 않은 기능을 미리 만들지 않는다.
2. 설명보다 결과를 우선한다.
3. 비개발자가 바로 실행 가능한 형태로 답한다.
4. 질문에는 질문만 답한다. 코드 수정이나 검증을 함께 보여주지 않는다.
5. 민감정보(아이디/비밀번호/API키)는 코드에 넣지 않는다.
6. 모든 주요 실행은 로그를 남긴다.
7. 실패 원인을 추적할 수 있어야 한다.
8. 자동화 기능에는 반드시 수동 실행 경로가 있어야 한다.

---

## 9. 금지사항

1. 한 파일에 여러 책임을 몰아넣는다.
2. 단계 밖 기능을 미리 만든다.
3. 민감정보를 하드코딩한다.
4. 도매처별 예외를 공통 파일에 누적한다.
5. 설명을 길게 늘려 토큰을 낭비한다.
6. 질문 답변 외 불필요한 코드/검증을 함께 출력한다.

---

## 11. 옵션 분류 규칙

- **옵션 추가금 있음**: `option_diffs`에 0이 아닌 값이 하나라도 있을 때
- **옵션 추가금 없음**: `option_diffs`가 없거나, 모든 값이 0일 때 (`"0\n0\n0"` 포함)
- 탭 필터 쿼리와 `_has_extra_price()` 함수는 반드시 동일한 기준을 사용해야 한다.
- 수집기는 절대가격을 저장하지 않는다. 기준가 대비 차액(0, +500, -100 형태)으로 변환해서 저장한다.

---

## 10. 보안 규칙

1. 아이디/비밀번호/쿠키/토큰/API 키는 코드에 넣지 않는다.
2. 환경변수(.env)를 사용한다.
3. 실제 `.env`는 커밋하지 않는다.
4. 로그에 민감정보를 남기지 않는다.

---

## 12. 진단 깊이 원칙

1. 표면 가설로 빠르게 답하지 않는다. 같은 문제가 2회 이상 반복되면 기존에 본 것 외의 새 가설(중복 데이터, 화면–DB 불일치, 카테고리/상태 분기 등)을 적극 탐색한다.
2. 사용자가 화면에서 보여준 ID(channel_no, origin_no, seller_code, signal_id, 시그널 ID 등)는 그 ID 그대로 사용한다.
3. 식별자 검색은 항상 중복 가능성을 점검한다. `.first()` 대신 `.all()`로 한 번 보고 시작한다.
4. 가설은 PUT/실행 전에 GET dump·로그·DB 조회로 사실 검증한 뒤 답변한다.
5. "빠르게 답하기"보다 "한 번에 정확하게 답하기"를 우선한다.

---

## 13. 답변 스타일 (비개발자 우선)

1. 답변의 기본 형식: 현재 어떤 상황인지 → 어떻게 처리할지 → 사용자가 직접 할 일.
2. 코드/스택 트레이스/필드명/JSON 등 기술 산출물은 답변 본문에 풀어 쓰지 않는다. 필요하면 코드 블록 안에 한정한다.
3. 검증/테스트 결과는 한 줄 요약(통과/실패)으로 압축한다. 단계별 출력은 노출하지 않는다.
4. "무엇을 고쳤다"보다 "무엇이 달라졌고 → 사용자는 어떻게 행동하면 된다"를 우선 전달한다.
5. 사용자가 직접 알아야 하는 부분(서버 재시작 여부, 어드민 처리 여부 등)을 항상 명시한다.
