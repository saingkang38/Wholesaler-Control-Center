# 설정 관리 정책 문서
버전: 1.0
상태: 운영 기준 확정본

## 문서 목적
이 문서는 어떤 값을 코드에 둘지, 설정으로 뺄지, 시크릿으로 관리할지 정의한다.  
목표는 하드코딩 방지, 보안 유지, 운영 변경 비용 최소화다.

---

## 1. 분류 원칙

모든 값은 아래 4종 중 하나로 분류한다.

1. 코드 상수
2. 일반 설정값
3. 시크릿 값
4. DB 저장 운영값

판단 없이 값을 추가하지 않는다.

---

## 2. 코드 상수

### 정의
자주 바뀌지 않고 시스템 구조상 고정된 값

### 예시
- 내부 enum 이름
- 기본 역할 이름(`admin`, `staff`)
- 공통 디렉토리 이름
- 변하지 않는 내부 키 이름
- 내부 상태 코드명

### 원칙
- 코드에 둘 수 있다.
- 의미 있는 상수 파일로 분리한다.
- 매직 문자열 하드코딩은 금지한다.

---

## 3. 일반 설정값

### 정의
민감하지 않지만 운영 중 바뀔 수 있는 값

### 예시
- 자동 실행 시간
- 기본 타임존
- 페이지 사이즈
- 기본 다운로드 파일명 규칙
- 도매처 활성/비활성
- 기본 비교 옵션
- UI 기본 필터값
- 재시도 횟수
- 타임아웃
- 기본 저장 경로

### 저장 위치 후보
- settings 파일
- DB의 settings 계층
- 관리자 설정 화면

### 원칙
- 코드 하드코딩 금지
- 가능하면 운영 중 변경 가능하게 설계
- 환경별 차이가 있으면 설정 분리 우선

---

## 4. 시크릿 값

### 정의
노출되면 안 되는 값

### 예시
- 관리자 초기 비밀번호
- 도매처 로그인 ID/PW
- 세션 secret key
- API 키
- 토큰
- 쿠키
- DB 접속 비밀번호
- 외부 서비스 인증키

### 저장 위치
- 환경변수
- 배포 플랫폼 시크릿 관리
- 필요 시 별도 암호화 저장

### 원칙
- 코드 저장소 커밋 금지
- 로그 출력 금지
- 화면 표시 금지
- `.env.example`에는 키 이름만 둔다
- 평문 저장 금지

---

## 5. DB 저장 운영값

### 정의
운영자가 변경하거나 이력으로 남겨야 하는 값

### 예시
- 도매처 사용 여부
- 마지막 실행 시간
- 사용자 메모
- 컬럼 매핑 규칙
- 커스텀 export 포맷 설정
- 기준 파일 이력
- 비교 기본 프로필
- 최근 실행 결과
- 운영 메모

### 원칙
- 운영 데이터는 DB에 저장한다.
- 코드나 정적 설정 파일에 넣지 않는다.
- 변경 이력이 필요한 값은 DB 우선

---

## 6. 절대 코드에 넣지 않는 값

1. 실제 도매처 계정 ID/PW
2. 실제 관리자 비밀번호
3. 세션 secret key
4. 배포 토큰
5. 알림 토큰
6. 실제 쿠키값
7. DB 실제 접속 비밀번호
8. 외부 서비스 인증키
9. 운영용 실계정 정보

---

## 7. `.env.example` 정책

프로젝트에는 실제 `.env`가 아니라 `.env.example`만 포함한다.

### 예시 키
- APP_ENV=
- SECRET_KEY=
- DATABASE_URL=
- ADMIN_INIT_USERNAME=
- ADMIN_INIT_PASSWORD=
- OWNERCLAN_LOGIN_ID=
- OWNERCLAN_LOGIN_PASSWORD=
- SCHEDULE_DAILY_TIME=
- DEFAULT_TIMEZONE=

### 원칙
- 실제 값 금지
- 키 이름만 제공
- 필요 시 짧은 주석만 허용
- 설명이 길어져 토큰을 늘리지 않는다

---

## 8. 환경별 분리

### local
개발 및 로컬 테스트

### staging
배포 전 점검

### production
실사용 환경

### 원칙
1. 환경마다 값이 달라질 수 있다.
2. 코드 분기보다 설정 분리를 우선한다.
3. 환경 차이를 코드 수정으로 해결하지 않는다.
4. 운영 문제를 긴급 하드코딩으로 해결하지 않는다.

---

## 9. 도매처 설정 정책

도매처별로 아래 항목은 설정화 가능성을 고려한다.

- 사용 여부
- 로그인 필요 여부
- 페이지 시작 번호
- 페이지 크기
- 재시도 횟수
- 상세 수집 여부
- 딜레이
- 타임아웃

### 1차 제한
도매처별 selector/XPath 전체를 관리자 화면에서 수정하는 구조는 하지 않는다.  
그 부분은 collector 코드 책임으로 둔다.

---

## 10. 비교 설정 정책

설정화 후보:
- 기본 비교 기준
- 품절 문자열 매핑
- 단종 문자열 매핑
- 상태 우선순위
- 무시할 컬럼 목록

### 원칙
1차에서는 꼭 필요한 일부만 설정화한다.  
복잡한 규칙은 compare rule 파일로 유지한다.

---

## 11. 출력 설정 정책

출력 설정은 아래로 나눈다.

### 구조 설정
- 컬럼 순서
- 헤더 이름
- 시트 구조

### 실행 설정
- 파일명 규칙
- 시트명
- 날짜 포맷

### 원칙
- 자주 바뀌는 값은 설정/DB로 분리
- 복잡한 가공 로직은 exporter 코드에 둔다
- 포맷별 예외를 공통 파일에 누적하지 않는다

---

## 12. Claude 규칙

새 값을 추가할 때 먼저 아래를 판단한다.

1. 코드 상수인가
2. 일반 설정값인가
3. 시크릿 값인가
4. DB 저장 운영값인가

이 판단 없이 하드코딩하지 않는다.

---

## 13. 점검 체크리스트

값을 추가할 때 아래를 확인한다.

- 자주 바뀌는가
- 민감한가
- 운영자가 바꿔야 하는가
- 환경마다 달라지는가
- 로그에 찍히면 안 되는가
- 코드에 두면 나중에 수정 비용이 커지는가

---

## 14. 초기 운영 권장안

1차 MVP 권장 방식:

- 시크릿 값 → 환경변수
- 운영 이력 / 사용자 데이터 → DB
- 자주 바뀌는 단순 옵션 → settings 계층
- 복잡한 업무 규칙 → 코드 또는 rule 파일

---

## 15. 비용과 안정성 원칙

1. 일부 비용이 들더라도 안정적인 저장/운영 방식을 우선 검토한다.
2. 시크릿 관리, DB 운영, 스케줄 실행은 무료보다 안정성을 우선한다.
3. 단, 1차 MVP에서는 과한 인프라 복잡도를 피한다.
4. 운영 리스크가 큰 부분은 단순한 구조보다 안전한 구조를 우선한다.

---

## 16. 금지사항

1. 민감정보 하드코딩
2. 실제 `.env` 커밋
3. 운영자가 바꿔야 할 값을 코드에 고정
4. 로그에 시크릿 출력
5. 환경 차이를 조건문 남발로 처리
6. 긴급 대응을 이유로 설정 정책 무시
