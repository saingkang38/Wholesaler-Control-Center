# 프로젝트명
도매처 통합 관리 시스템

## 문서 목적
이 문서는 Claude Code의 작업 방식, 응답 방식, 금지사항을 정의한다.
**Claude Code가 실제로 덜 헤매고 덜 읽고 정확하게 일하게 만드는 운영 규칙**이다.

---

## 1. 프로그램 목적 (최우선 기준)

1. 도매처를 연결해 상품정보를 수집한다.
2. 수집한 상품을 스마트스토어와 매칭한다.
3. 도매처의 변동사항(가격/이미지/옵션/상태 등 모든 항목)을 감지하여 알린다.
4. 변동사항은 프로그램 내에서 직접 스마트스토어에 반영할 수 있어야 한다.
5. 모든 변경 내용은 직관적으로 화면에서 확인할 수 있어야 한다.
6. 수집/처리 진행 과정을 실시간으로 볼 수 있어야 한다.
7. 모든 작업 기록은 남긴다.
8. 각 변동사항은 개별적으로 수정 가능하며, 한 항목 수정 시 다른 항목은 절대 변경되지 않는다.
9. 한 상품에 가격변동/이미지변동이 동시에 있어도, 각각 별도 작업으로 처리한다.

---

## 2. 구조 원칙 (모듈형 필수)

1. 모든 기능은 모듈형으로 만든다. 나중에 추가/교체/삭제가 가능해야 한다.
2. 수집 / 정규화 / 마스터 / 액션 / 스토어 / 인증 / 로그를 분리한다.
3. 도매처별 수집 로직은 `collectors/{wholesaler_name}/` 구조를 따른다.
4. 단일 파일에 여러 책임을 몰아넣지 않는다.
5. 도매처별 예외를 공통 파일에 누적하지 않는다.
6. 공통 계층 수정은 예외적으로만 허용한다.

---

## 3. 변동감지 기준

수집 항목 중 아래 모든 항목에 대해 변동을 감지하고 액션 시그널을 생성한다.

- 가격 변동 (인상 필요 / 인하 가능)
- 상태 변동 (품절 / 단종 / 재입고)
- 옵션가 변동 / 옵션 구성 추가·변경
- 이미지 변동 (감지만, 반영은 수동)
- 상품명 변동 (감지만, 반영은 수동)
- 배송비/배송조건 변동 (감지만, 반영은 수동)

각 변동은 독립적인 시그널로 생성되며, 하나를 처리해도 다른 시그널에 영향을 주지 않는다.

---

## 4. 액션 시그널 유형

| 시그널 | 설명 | 처리 방식 |
|--------|------|-----------|
| PRICE_UP_NEEDED | 도매가 인상 → 판매가 올려야 함 | 자동 반영 가능 |
| PRICE_DOWN_POSSIBLE | 도매가 인하 → 판매가 내릴 수 있음 | 자동 반영 가능 |
| SUSPEND_NEEDED | 도매처 품절/단종 → 판매 중지 필요 | 자동 반영 가능 |
| RESUME_POSSIBLE | 도매처 재입고 → 판매 재개 가능 | 자동 반영 가능 |
| DISCONTINUE_NEEDED | 장기 품절 → 단종 처리 필요 | 자동 반영 가능 |
| OPTION_PRICE_CHANGE | 옵션 가격 차액 변동 | 자동 반영 가능 |
| OPTION_ADD | 도매처 옵션 구성 변경 → 전체 교체 | 자동 반영 가능 |
| OPTION_STOCK_CHANGE | 옵션별 재고 변동 | 자동 반영 가능 |

**처리 규칙:**
- PRICE_UP/DOWN 실행 시 옵션 상품이면 옵션추가금도 함께 갱신, 이후 OPTION_PRICE_CHANGE 자동 스킵
- OPTION_ADD는 도매처 옵션으로 무조건 전체 교체 (부분 추가 없음)
- 옵션 있는 상품의 가격 변경은 반드시 정가+즉시할인+옵션추가금 세트로 업데이트

## 6. 옵션 표준화 규칙

도매처마다 옵션 형식이 다르므로, 수집 시 아래 키로 통일한다.

- `extra["옵션"]` : 옵션명 (줄바꿈 구분 문자열)
- `extra["옵션가"]` : 기준가 대비 차액 (줄바꿈 구분 문자열, 예: `"0\n500\n-100"`)

리스트 형식은 허용하지 않는다. 절대가격이 수집되면 기준가 대비 차액으로 변환해서 저장한다.

**책임 위치:** 표준화는 각 수집기(collectors/{code}/)에서 직접 수행한다. 정규화 레이어는 문자열 포맷 검증만 담당한다.
