# 프로젝트명
도매처 통합 관리 시스템

## 문서 목적
이 문서는 Claude Code의 작업 방식, 응답 방식, 금지사항을 정의한다.
**Claude Code가 실제로 덜 헤매고 덜 읽고 정확하게 일하게 만드는 운영 규칙**이다.

---

## 1. 프로그램 목적 (최우선 기준)

1. 도매처를 연결해 상품정보를 수집한다.
2. 수집한 상품을 스마트스토어와 매칭한다.
3. 도매처의 변동사항(가격/이미지/옵션/상태 등 모든 항목)을 감지하여 알린다.
4. 변동사항은 프로그램 내에서 직접 스마트스토어에 반영할 수 있어야 한다.
5. 모든 변경 내용은 직관적으로 화면에서 확인할 수 있어야 한다.
6. 수집/처리 진행 과정을 실시간으로 볼 수 있어야 한다.
7. 모든 작업 기록은 남긴다.
8. 각 변동사항은 개별적으로 수정 가능하며, 한 항목 수정 시 다른 항목은 절대 변경되지 않는다.
9. 한 상품에 가격변동/이미지변동이 동시에 있어도, 각각 별도 작업으로 처리한다.

---

## 2. 구조 원칙 (모듈형 필수)

1. 모든 기능은 모듈형으로 만든다. 나중에 추가/교체/삭제가 가능해야 한다.
2. 수집 / 정규화 / 마스터 / 액션 / 스토어 / 인증 / 로그를 분리한다.
3. 도매처별 수집 로직은 `collectors/{wholesaler_name}/` 구조를 따른다.
4. 단일 파일에 여러 책임을 몰아넣지 않는다.
5. 도매처별 예외를 공통 파일에 누적하지 않는다.
6. 공통 계층 수정은 예외적으로만 허용한다.

---

## 3. 변동감지 기준

수집 항목 중 아래 모든 항목에 대해 변동을 감지하고 액션 시그널을 생성한다.

- 가격 변동 (인상 필요 / 인하 가능)
- 상태 변동 (품절 / 단종 / 재입고)
- 옵션가 변동 / 옵션 구성 추가·변경
- 이미지 변동 (감지만, 반영은 수동)
- 상품명 변동 (감지만, 반영은 수동)
- 배송비/배송조건 변동 (감지만, 반영은 수동)

각 변동은 독립적인 시그널로 생성되며, 하나를 처리해도 다른 시그널에 영향을 주지 않는다.

---

## 4. 액션 시그널 유형

| 시그널 | 설명 | 처리 방식 |
|--------|------|-----------|
| PRICE_UP_NEEDED | 도매가 인상 → 판매가 올려야 함 | 자동 반영 가능 |
| PRICE_DOWN_POSSIBLE | 도매가 인하 → 판매가 내릴 수 있음 | 자동 반영 가능 |
| SUSPEND_NEEDED | 도매처 품절/단종 → 판매 중지 필요 | 자동 반영 가능 |
| RESUME_POSSIBLE | 도매처 재입고 → 판매 재개 가능 | 자동 반영 가능 |
| DISCONTINUE_NEEDED | 장기 품절 → 단종 처리 필요 | 자동 반영 가능 |
| OPTION_PRICE_CHANGE | 옵션 가격 차액 변동 | 자동 반영 가능 |
| OPTION_ADD | 도매처 옵션 구성 변경 → 전체 교체 | 자동 반영 가능 |
| OPTION_STOCK_CHANGE | 옵션별 재고 변동 | 자동 반영 가능 |

**처리 규칙:**
- PRICE_UP/DOWN 실행 시 옵션 상품이면 옵션추가금도 함께 갱신, 이후 OPTION_PRICE_CHANGE 자동 스킵
- OPTION_ADD는 도매처 옵션으로 무조건 전체 교체 (부분 추가 없음)
- 옵션 있는 상품의 가격 변경은 반드시 정가+즉시할인+옵션추가금 세트로 업데이트

## 6. 옵션 표준화 규칙

도매처마다 옵션 형식이 다르므로, 수집 시 아래 키로 통일한다.

- `extra["옵션"]` : 옵션명 (줄바꿈 구분 문자열)
- `extra["옵션가"]` : 기준가 대비 차액 (줄바꿈 구분 문자열, 예: `"0\n500\n-100"`)

리스트 형식은 허용하지 않는다. 절대가격이 수집되면 기준가 대비 차액으로 변환해서 저장한다.

**책임 위치:** 표준화는 각 수집기(collectors/{code}/)에서 직접 수행한다. 정규화 레이어는 문자열 포맷 검증만 담당한다.

## 9. 반자동 검수 원칙

이 프로그램은 "자동으로 바로 수정하는 프로그램"이 아니라,
**"이상이나 변경사항을 감지하고 액션관리에서 검수 후 승인하는 반자동 프로그램"**이다.

### 9-1. 핵심 원칙

1. 자동화보다 정확도를 우선한다.
2. 최종 방식은 반자동이다.
3. "신호 생성 → 액션관리 검수 → 승인 후 반영" 흐름이 기본.
4. 코드가 판단해서 바로 반영하는 방향으로 확장하지 않는다.

### 9-2. B 정책 (전부 신호 생성 + 검수 후 반영) 22개 항목

아래 22개는 **전부 자동 반영 금지, 신호만 생성** 원칙을 따른다.

| # | 항목 |
|:--:|------|
| 1 | 단품 가격 인상 |
| 2 | 단품 가격 인하 |
| 3 | 옵션 상품 가격 인상 |
| 4 | 옵션 상품 가격 인하 |
| 5 | 세일 상품 가격 변경 |
| 6 | 옵션 추가금 변경 |
| 7 | 옵션 재고 변경 |
| 8 | 옵션명 변경 |
| 9 | 옵션 추가 |
| 10 | 옵션 삭제 |
| 11 | 옵션 전체 교체 |
| 12 | 품절 처리 |
| 13 | 장기 품절 처리 |
| 14 | 미수집 처리 |
| 15 | 단종 후보 처리 |
| 16 | 단종 확정 처리 |
| 17 | 재입고 복구 |
| 18 | 단종 후 복구 |
| 19 | 매칭 실패 |
| 20 | 도매가 0원 / NULL |
| 21 | 데이터 이상치 |
| 22 | CLOSE 상태 복구 |

### 9-3. 품단종 상태 5개 구분

상품 상태는 반드시 아래 5개로 구분한다.

| 상태 | 코드값 | 의미 | 전이 기준 |
|------|--------|------|-----------|
| 정상 판매중 | `active` | 정상 수집·판매 | 기본 |
| 품절 | `out_of_stock` | 도매처에서 품절로 수집 | 도매처 표기 |
| 일시적으로 수집 안 됨 | `missing` | 1일 이상 미수집 | `missing_days >= 1` |
| 단종 후보 | `discontinued_candidate` | 60일 이상 미수집 | `missing_days >= 60` |
| 단종 확정 | `discontinued` | 365일 이상 미수집 | `missing_days >= 365` |

- 품절 상태에서도 **missing_days를 함께 누적**한다.
- 단종 확정된 상품은 **자동 복구하지 말고 수동 복구만 허용**.

### 9-4. 가격 판단 기준

- 가격 판단의 기본 기준은 **도매가**.
- 세일/즉시할인 상품도 신호 생성은 가능하되, 즉시 반영은 금지.
- 도매처의 가격 체계가 소비자가/세일가/기업회원가로 나뉘는 경우 **수집값의 출처**를 섹션 10에 명시한다.

### 9-5. 옵션 처리 철학

- 옵션 상품을 단품과 별도 철학으로 보지 않는다.
- **무조건 도매처 기준**으로 스마트스토어를 맞춘다.
- 도매처 옵션이 바뀌면 우리도 동일하게 재구성. 단, 자동 반영 금지.
- 옵션 수 불일치 / 옵션명 변경 / 추가금 변동 모두 신호 생성 후 검수.
- 옵션이 자주 바뀌는 상품도 계속 자동 추적 대상이다.

### 9-6. 예외 처리 원칙

- 스토어 상품 있는데 도매처 매칭 안 됨 → 별도 **미매칭 검수 공간**에 노출.
- 도매가 0/NULL → 무조건 제외 말고 **검수 대상으로 노출**.
- 데이터 이상치(옵션 깨짐·가격 누락·재고 이상) → 경고 후 수동 검수.
- CLOSE 등 자동 복구 어려운 상태 → 별도 **수동 처리 목록**에 분리.

## 10. 도매처별 가격/옵션 계산 정책

각 도매처마다 수집되는 가격 필드·옵션가 형식이 다를 수 있다.
이 섹션은 **실제 코드가 참조할 기술 정책**을 기록한다.

> **작성 형식 (각 도매처 공통)**
> - 기준 가격 필드: 어느 필드를 `master.price` 로 매핑하는가
> - 옵션가 형식: 차액(기준가 대비 ±) / 절대가격 / 혼합
> - 부가세 처리: 포함 / 별도 / 불명
> - 세일가 필드: 있는가 / 수집 중인가 / 무시인가
> - 기업회원가: 있는가 / 수집 중인가
> - 특이사항: 기타 코드 수정 시 고려할 사항

> **주의**: 이 섹션의 각 항목은 **초기 미확정 상태**이며, 실제 검증 완료 후 사실만 기록한다. 추측은 넣지 않는다.
