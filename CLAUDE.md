# 프로젝트명
도매처 통합 관리 시스템

## 문서 목적
이 문서는 Claude Code의 작업 방식, 응답 방식, 금지사항을 정의한다.  
프로젝트 설명은 `PROJECT_PRD.md`를 따르고, 단계는 `PHASE_PLAN.md`를 따른다.

이 문서는 길게 설명하는 문서가 아니라,  
**Claude Code가 실제로 덜 헤매고 덜 읽고 정확하게 일하게 만드는 운영 규칙**이다.

---

## 1. 최우선 원칙

1. 이 프로젝트는 프로그램 완성이 최우선이다.
2. 한 번에 전체를 만들지 않고 단계별로 진행한다.
3. 설명보다 결과를 우선한다.
4. 비개발자가 바로 실행 가능한 형태로 답한다.
5. 토큰 낭비를 줄이기 위해 현재 요청 범위만 다룬다.
6. 모듈형 구조를 유지한다.
7. 공식 배포 문서로 확인된 Claude Code 지침만 반영한다.
8. 베타, 실험 기능, 커뮤니티 추측, 불확실한 팁은 반영하지 않는다.
9. 민감정보는 코드에 넣지 않는다.
10. 1차는 오너클랜 1개 도매처 파일럿 성공이 최우선이다.

---

## 2. Claude 작업 순서

Claude는 항상 아래 순서로 작업한다.

1. 현재 요청이 어느 Phase 범위인지 판단
2. 현재 단계에서 필요한 파일 범위만 지정
3. 범위 밖 작업 제외
4. 사용자가 해야 할 일과 Claude가 할 일을 분리
5. 필요한 결과물 작성
6. 확인 방법 제시

---

## 3. 응답 규칙

1. 설명은 최소화한다.
2. 긴 배경 설명을 하지 않는다.
3. 질문은 꼭 필요한 경우에만 한다.
4. 가능하면 파일 단위 완성본으로 제안한다.
5. 사용자가 요청하지 않은 과도한 확장 구현을 하지 않는다.
6. 한 번에 한 단계만 진행한다.
7. 결과물 중심으로 답한다.
8. 사용자가 복붙할 수 있는 형태를 우선한다.

---

## 4. 비개발자용 기본 출력 형식

가능하면 아래 형식을 사용한다.

1. 현재 단계
2. 이번 작업 목표
3. Claude가 할 일
4. 사용자가 할 일
5. 사용자가 실행 후 보내줘야 할 결과
6. 완료 기준
7. 복붙용 결과물

불필요한 이론 설명은 줄인다.

---

## 5. 토큰 절약 규칙

1. 현재 요청과 직접 관련된 파일만 우선 본다.
2. 전체 코드베이스 재탐색을 기본값으로 하지 않는다.
3. 같은 설명을 반복하지 않는다.
4. 다음 단계 구현을 미리 하지 않는다.
5. 사용자가 요청하지 않은 대안 여러 개를 길게 늘어놓지 않는다.
6. 작은 변경은 작은 범위에서 끝낸다.
7. 이미 확정된 문서를 다시 장황하게 재서술하지 않는다.
8. 긴 설명보다 완성본을 우선 제공한다.

---

## 6. 구조 원칙

1. 단일 거대 파일 구조로 개발하지 않는다.
2. 수집 / 정규화 / 비교 / 출력 / 인증 / 로그를 분리한다.
3. 도매처별 수집 로직은 `collectors/{wholesaler_name}/` 구조를 따른다.
4. 비교 로직은 compare 계층에 둔다.
5. 출력 양식은 exporter 계층에 둔다.
6. 인증/사용자 로직은 collector나 exporter에 섞지 않는다.
7. 공통 계층 수정은 예외적으로만 허용한다.

---

## 7. 구현 원칙

1. 먼저 1개 도매처 파일럿을 완성하고 이후 확장한다.
2. 먼저 내부 구조와 실행 안정성을 맞추고 UI는 후순위로 둔다.
3. 수집 결과는 raw → normalize → save → compare → export 순서를 따른다.
4. 비교 기준은 기본적으로 `도매처 + 상품코드` 조합을 사용한다.
5. 자동화 기능에는 반드시 수동 실행 경로가 있어야 한다.
6. 모든 주요 실행은 로그를 남겨야 한다.
7. 실패 원인을 추적할 수 있어야 한다.

---

## 8. 문서 사용 규칙

- 프로젝트 목표/범위: `PROJECT_PRD.md`
- 개발 단계/순서: `PHASE_PLAN.md`
- 모듈 책임 경계: `MODULE_BOUNDARIES.md`
- 데이터 구조: `DATA_SCHEMA_DRAFT.md`
- 설정/시크릿 정책: `CONFIG_POLICY.md`
- 변경 대응 기준: `CHANGE_STRATEGY.md`

이 문서에 위 내용을 길게 중복 작성하지 않는다.

---

## 9. 금지사항

1. 기존 꼬인 구조를 그대로 답습하지 않는다.
2. 한 파일에 여러 책임을 몰아넣지 않는다.
3. 도매처별 예외를 공통 파일에 누적하지 않는다.
4. 단계 밖 기능을 미리 만들지 않는다.
5. 민감정보를 하드코딩하지 않는다.
6. 비교 규칙을 UI 코드나 화면 로직에 넣지 않는다.
7. 출력 포맷 분기를 한 파일에 계속 누적하지 않는다.
8. 설명을 길게 늘려 토큰을 낭비하지 않는다.
9. 공식 문서로 확인되지 않은 Claude Code 사용법을 규칙처럼 반영하지 않는다.

---

## 10. 보안 규칙

1. 아이디/비밀번호/쿠키/토큰/API 키는 코드에 넣지 않는다.
2. 환경변수 또는 안전한 시크릿 저장 방식을 사용한다.
3. 로그에 민감정보를 남기지 않는다.
4. 실제 `.env`는 커밋하지 않는다.
5. 계정 정보 평문 저장을 금지한다.

---

## 11. 변경 요청 처리 규칙

변경 요청을 받으면 가능하면 아래 형식으로 답한다.

1. 변경 분류
2. 수정 범위
3. 수정 대상 파일
4. 사용자가 할 일
5. 결과물
6. 확인 방법

---

## 12. 1차 파일럿 대상

- 오너클랜

기존 구현은 참고만 가능하다.  
구조는 현재 문서 기준으로 다시 잡는다.

---

## 13. 최종 행동 기준

1. 문서와 현재 단계 범위를 먼저 본다.
2. 현재 요청에 필요한 최소 범위만 수정한다.
3. 비개발자가 실행 가능한 결과물을 준다.
4. 결과 확인 방법까지 준다.
5. 다음 단계는 현재 단계 완료 후 진행한다.
